# Pre-Coding Guardrails

Nine lightweight but critical guardrails to run **before** writing a single line of code. They exist because fixing mistakes retroactively is expensive; a 30-second check now saves hours later.

## Guardrail 1 — Lessons learned (no repeat mistakes)

Read `lessons-learned/lessons-learned.md` before starting. For each entry relevant to the current task (same file area, same pattern, same component type), surface it explicitly:

> ⚠️ **Lessons-learned hit**: [entry title] — [one-line reminder of the mistake and the fix]

After finishing, if a new anti-pattern emerged during the work, **propose an addition** to the log — don't wait for the next person to rediscover it.

## Guardrail 2 — Naming conventions

Before creating any file or symbol, verify against the naming table in the matching `stacks/` profile: PascalCase component folders matching the export, `use*` hooks, verb+entity API functions, `is/has/can` booleans, BEM kebab-case SCSS, no default exports, and — critically — **no duplicate of an existing shared component** (grep first).

## Guardrail 3 — Scope boundary (ask before crossing)

1. **Primary scope** = the folders the task/ticket names.
2. **Implicit scope** = direct dependencies — fine to *read*, editing crosses the line.
3. **Out of scope** = `shared/`, `app/`, other modules, root configs.

Before touching any out-of-scope file, pause and ask:

> 🚧 **Scope check**: about to modify `[path]`, outside primary scope (`[scope]`). Reason: [one sentence]. Proceed?

Never silently edit shared code, even for a "trivial" barrel export. Exception: the task explicitly names the shared file.

## Guardrail 4 — Design tokens

For any styling change, verify every color/spacing/font-size/radius/shadow value is a token (`var(--…)`) or project mixin. No exact token → closest semantic token → still nothing → **STOP and ask** with candidates listed. Full procedure in `stacks/react-typescript.md (Styling)`.

## Guardrail 5 — Data contracts

Any fetcher behind a table: `page` is **0-based**, return shape is `{ data, total }`, slicing is `start = page * pageSize`. Clone the canonical reference implementation rather than writing from scratch. Smoke-test pages 1 → 2 → 3 (first row id must change). All traps in `stacks/react-typescript.md (Silent-failure traps)`.

## Guardrail 6 — Plan self-review (every plan, every time)

Every plan — first or fifth of the session — is checked against all nine guardrails **before** being presented. Found an issue? Fix the plan; never present a plan with known problems for the user to catch. Running the guardrails once at session start does not cover later plans.

## Guardrail 7 — Duplication pass (on review tasks)

When the task is reviewing a file/module, do a dedicated duplication pass after reading: copy-paste blocks, repeated literals, parallel functions differing by one parameter, repeated JSX. For each real duplication, either propose a concrete extraction (respecting Guardrail 3) or explicitly warn — never stay silent. Don't force-merge incidental look-alikes; premature DRY is its own anti-pattern.

## Guardrail 8 — Plan-gated execution

Prompt asks for a plan (any language) → produce the plan and **STOP**. Read-only investigation is allowed; mutations are not. Wait for explicit approval; ambiguous replies get a question. Never bundle plan + execution in one turn.

## Guardrail 9 — No ticket / scaffolding comments

Never add — and remove on sight, within scope: ticket IDs, design-tool node references, `old`/`new` migration markers, commented-out old implementations, "generated by / copied from" breadcrumbs. Git and the ticket system already track provenance. Keep only comments that explain non-obvious logic to a reader who has never seen the ticket.

## Quick start (run at the beginning of every task)

```
1. Read lessons-learned.md → surface hits
2. Identify primary scope folder(s)
3. List files the plan touches → flag out-of-scope ones
4. Verify naming for every new file/symbol
5. Styling → tokens only; ask before new tokens
6. Table fetchers → 0-based page, { data, total }
7. Review task → duplication pass, never silent
8. Plan requested → plan ONLY, stop, wait for approval
9. No ticket/design/migration comments
10. Self-review the plan against all guardrails → improve first
11. Implement only after approval
12. Done → propose lessons-learned additions
```
